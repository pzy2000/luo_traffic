# Legacy Road Quality Root Cause

本文说明之前车辆模拟地图质量差的主要原因，以及为什么这次要改成严格 OSM 拓扑。

## 旧问题一：视觉道路和车辆路由图不是同一份对象

旧实现中，页面可见道路和车辆实际行驶图可能来自不同构建链路：

- 视觉道路来自 GDAL/OGR 或 Overpass 几何。
- 车辆路由图由另一个预计算脚本重新生成。

这样会导致车辆看起来贴着路跑，但真实拓扑和屏幕上的线并不完全一致。车辆经过复杂路口、立交、并行道路时，很容易出现跳线、抖动、错误转向。

## 旧问题二：使用几何吸附修补连通性

旧图为了提升连通率，生成了大量合成边：

- short connector
- short extension
- short junction
- endpoint-to-segment repair

这些方法对普通平面路口有时能补洞，但在上海这种高架、匝道、隧道、辅路密集的环境中风险很高。

典型错误：

- 高架主线和地面道路二维投影很近，被短线接上。
- 平行辅路和主路端点靠近，被误认为可互通。
- 匝道附近多个端点密集，几何修补把不该连的层级接在一起。
- 车辆在桥上或隧道口附近突然转向地面路。

## 旧问题三：OSM node refs 没有成为唯一拓扑依据

OSM 道路的真实连通关系来自 way 的 node refs。旧实现有些步骤只保留了 polyline 几何，却丢失或弱化了原始 node id 语义。

一旦只按坐标或线段距离推断连通性，就无法可靠区分：

- 真实路口。
- 立交桥下穿。
- 隧道穿越。
- 不同 layer 的交叉。
- 主路与辅路之间没有匝道的并行关系。

## 旧问题四：桥隧和 layer 语义没有完整进入车辆算法

即使数据里有 `bridge`、`tunnel`、`layer`，如果 compact graph hydrate 后没有恢复这些字段，浏览器端车辆算法仍然无法知道某条边是高架、桥、隧道还是普通地面路。

旧算法只能凭 highway rank 和二维方向猜测，复杂立交场景下很难避免错误。

## 这次修复的核心

这次重建采用“宁可少连，也不误连”的原则：

- 同一批裁切 roads 同时用于底图和车辆路由。
- 车辆图只按 OSM node refs 连接。
- 关闭 connector。
- 关闭 geometry repair。
- 保留 `bridge` / `tunnel` / `layer`。
- 标记 `boundaryClip`，区分闵行裁切边界死路和区内死路。
- 车辆转向统一经过 `isLegalTrafficTransition(...)`。

因此质量提升不是因为把线画粗或视觉上修得更漂亮，而是因为车辆实际使用的拓扑已经接近路由引擎使用 OSM 的方式：真实共享节点才连，不同层级投影交叉不连。
